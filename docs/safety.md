# Safety Model

Quasar uses `unsafe` for zero-copy account access, raw CPI syscalls, pointer arithmetic over the SVM input buffer, and `MaybeUninit` patterns for stack-allocated arrays. This document catalogs every unsafe usage, explains the soundness argument for each pattern category, and describes the compile-time and runtime protections against common attack vectors.

## Unsafe Inventory

48 `unsafe` blocks and 12 `unsafe` trait impls / function signatures across `core/src/`. Grouped by pattern category:

### Zero-Copy Pointer Casts (`& -> &ZcType`)

| Location | Code | Purpose |
|----------|------|---------|
| `accounts/account.rs:154` | `&*(view as *const AccountView as *const Self)` | Cast `&AccountView` to `&Account<T>` (shared) |
| `accounts/account.rs:202` | `T::deref_from(&self.view)` | Deref to ZC companion struct via `ZeroCopyDeref` trait |
| `accounts/account.rs:210` | `T::deref_from_mut(&self.view)` | DerefMut to ZC companion struct |
| `accounts/initialize.rs:20` | `&*(view as *const AccountView as *const Self)` | Cast `&AccountView` to `&Initialize<T>` (shared) |
| `accounts/sysvar.rs:27` | `&*(view as *const AccountView as *const Self)` | Cast `&AccountView` to `&Sysvar<T>` |
| `accounts/sysvar.rs:37` | `T::from_bytes_unchecked(self.view.borrow_unchecked())` | Cast sysvar bytes to `&T` |
| `sysvars/mod.rs:36` | `&*(bytes.as_ptr() as *const Self)` | Generic `from_bytes_unchecked` for sysvar types |

**Soundness**: All these casts rely on three invariants:

1. **`#[repr(transparent)]`**: `Account<T>`, `Initialize<T>`, and `Sysvar<T>` are `#[repr(transparent)]` wrappers over `AccountView`. The cast is a no-op at the memory level.

2. **Alignment 1**: ZC companion structs (generated by `#[account]`) use Pod types (`PodU64`, `PodU128`, etc.) which are `#[repr(transparent)]` over `[u8; N]`. Compile-time assertions enforce `align_of::<T>() == 1` for every ZC struct. Since the source data (`&[u8]`) also has alignment 1, the pointer cast is always aligned.

3. **Bounds checked upstream**: `AccountCheck::check()` validates `data_len >= size_of::<ZcStruct>() + discriminator_len` before any pointer cast occurs. The `Deref` impl relies on this upstream check -- it performs no bounds validation itself.

### Interior Mutability (`& -> &mut` Casts)

| Location | Code | Purpose |
|----------|------|---------|
| `accounts/account.rs:172` | `&mut *(view as *const AccountView as *mut Self)` | Cast `&AccountView` to `&mut Account<T>` |
| `accounts/initialize.rs:36` | `&mut *(view as *const AccountView as *mut Self)` | Cast `&AccountView` to `&mut Initialize<T>` |
| `macros.rs:43` | Same pattern via `define_account!` macro | Cast for program-defined account types |

**Soundness**: This is the most controversial pattern in the framework. The argument:

1. `Account<T>` is `#[repr(transparent)]` over `AccountView`, which holds a raw `*mut RuntimeAccount` pointer.
2. All mutations (e.g., `set_lamports`, `assign`, writing to account data) go through `AccountView`'s raw pointer methods, not through the `&mut` reference itself.
3. The `&mut Account<T>` reference is never used to write to the `AccountView` struct's own memory -- it exists solely to provide `DerefMut` access to the ZC companion struct, which itself writes through the same raw pointers.
4. Under the Tree Borrows model, the `&mut` retag creates a "Reserved" permission. Since no write occurs through the mutable reference to `AccountView`'s own fields, the permission never transitions to "Active", and no UB occurs.

This pattern is UB under the Stacked Borrows model but sound under Tree Borrows. It is the same approach used by Pinocchio and Anchor's internal account handling. All `& -> &mut` cast sites are validated under Miri with `-Zmiri-tree-borrows`.

Every call site checks `is_writable()` before performing the cast, returning `ProgramError::Immutable` for read-only accounts.

### Raw CPI Syscalls

| Location | Code | Purpose |
|----------|------|---------|
| `cpi/mod.rs:35-56` | `RawCpiAccount::from_view` | Build CPI account struct from `AccountView` raw pointer |
| `cpi/mod.rs:49-53` | `copy_nonoverlapping` 3-byte flag extraction | Copy `is_signer`, `is_writable`, `executable` from `RuntimeAccount` |
| `cpi/mod.rs:73-106` | `invoke_raw` | Direct `sol_invoke_signed_c` syscall |
| `cpi/mod.rs:126-133` | `MaybeUninit` array init for `CpiCall::new` | Element-by-element init of `[RawCpiAccount; ACCTS]` |
| `cpi/mod.rs:160-170` | `invoke_inner` | Pass owned arrays to `invoke_raw` |
| `cpi/system.rs:24-31` | `MaybeUninit` + `copy_nonoverlapping` | Build 52-byte `create_account` instruction data |

**Soundness**: The CPI path is built around two compile-time assertions at `cpi/mod.rs:28-29`:

```rust
const _: () = assert!(core::mem::size_of::<RawCpiAccount>() == 56);
const _: () = assert!(core::mem::align_of::<RawCpiAccount>() == 8);
```

`RawCpiAccount` is `#[repr(C)]` with a layout matching the SVM's expected `CpiAccount` format. The 56-byte size assertion catches any field addition or reordering at compile time.

The 3-byte `copy_nonoverlapping` at `cpi/mod.rs:49-53` reads `RuntimeAccount` fields at offsets 1, 2, 3 (`is_signer`, `is_writable`, `executable`). This relies on `RuntimeAccount` being `#[repr(C)]` in the `solana-account-view` crate with `borrow_state: u8` at offset 0. The audit recommends compile-time `offset_of!` assertions to guard against upstream layout changes:

```rust
const _: () = assert!(core::mem::offset_of!(RuntimeAccount, is_signer) == 1);
```

The `MaybeUninit` array init in `CpiCall::new` uses a `while` loop that writes exactly `ACCTS` elements via `ptr::write`, then calls `assume_init`. This is sound because the const generic `ACCTS` bounds both the loop iteration count and the array size.

### Pointer Arithmetic (Remaining Accounts)

| Location | Code | Purpose |
|----------|------|---------|
| `remaining.rs:44` | `(*raw).borrow_state` | Read `RuntimeAccount` field via raw pointer |
| `remaining.rs:48` | `AccountView::new_unchecked(raw)` | Create `AccountView` from validated pointer |
| `remaining.rs:55-58` | `ptr.add(ACCOUNT_HEADER + data_len)` + alignment round-up | Advance past account header + data |
| `remaining.rs:61` | `ptr.add(size_of::<u64>())` | Skip 8-byte duplicate marker |
| `remaining.rs:98` | `ptr::read(declared.as_ptr().add(idx))` | Copy `AccountView` from declared accounts array |
| `remaining.rs:160` | `ptr::read(self.cache_ptr().add(remaining_idx))` | Read from `MaybeUninit` iterator cache |
| `remaining.rs:186-188` | Same advancement pattern as line 55-58 | Iterator `next()` pointer advancement |
| `remaining.rs:200-202` | `ptr::read` + `ptr::write` into cache | Cache `AccountView` for future dup resolution |

**Soundness**: All pointer arithmetic in `remaining.rs` depends on the SVM serialization format:

```
[account_count: u64]
[RuntimeAccount header | data | padding to 8-byte alignment] * N
[ix_data_len: u64]
[ix_data bytes]
[program_id: 32 bytes]
```

The boundary pointer (`accounts_boundary`) is computed as `ix_data.as_ptr().sub(8)` at `entrypoint.rs:36` -- pointing to the `ix_data_len` field, which marks the end of the accounts region. Every pointer dereference in the iterator is preceded by a boundary check: `ptr as *const u8 >= self.boundary`.

The alignment round-up `((ptr as usize + 7) & !7) as *mut u8` at `remaining.rs:58` involves an integer round-trip that loses pointer provenance under `-Zmiri-strict-provenance`. This is not UB under the default provenance model and matches the standard Solana deserialization pattern.

The `MaybeUninit<[AccountView; 64]>` cache at `remaining.rs:141` is only read at positions `0..self.index` (elements that were previously written). A bounds check at `remaining.rs:165` guards against forward-referencing duplicates:

```rust
if remaining_idx >= self.index {
    return None;
}
```

### Entrypoint Dispatch

| Location | Code | Purpose |
|----------|------|---------|
| `entrypoint.rs:10-12` | `&*(ix_data.as_ptr().add(ix_data.len()) as *const [u8; 32])` | Read program ID from SVM buffer past end of `ix_data` slice |
| `entrypoint.rs:13` | `($ptr as *mut u8).add(size_of::<u64>())` | Skip 8-byte account count header |
| `entrypoint.rs:18-19` | `*(ix_data.as_ptr() as *const [u8; N])` | Read discriminator bytes from instruction data |
| `entrypoint.rs:27-28` | `parse_accounts(__accounts_start, &mut __buf)` | Write `AccountView` values into `MaybeUninit` buffer |

| `entrypoint.rs:36` | `ix_data.as_ptr().sub(size_of::<u64>())` | Compute end-of-accounts boundary pointer |

**Soundness**: The program ID read at `entrypoint.rs:10-12` accesses 32 bytes immediately past the end of the `ix_data` slice. This is sound because the SVM places the program ID at that exact location in the serialized buffer -- both `ix_data` and the program ID are part of the same allocation. Validated under Miri with the `program_id_read_from_end_of_slice` test.

The discriminator read at line 18-19 is guarded by a length check at line 15: `if ix_data.len() < disc_len`. The cast from `*const u8` to `*const [u8; N]` has alignment 1 on both sides.

The `assume_init` at line 30 depends on the `parse_accounts` function (generated by `#[derive(Accounts)]`) initializing all `COUNT` elements. This is a delegated safety invariant -- a bug in the derive macro's codegen could leave uninitialized `AccountView` values.

### Sysvar Access

| Location | Code | Purpose |
|----------|------|---------|
| `sysvars/mod.rs:41-64` | `impl_sysvar_get!` macro | Syscall + `MaybeUninit` + `assume_init` |
| `sysvars/mod.rs:79-109` | `get_sysvar_unchecked` | Raw `sol_get_sysvar` syscall wrapper |

**Soundness**: The `impl_sysvar_get!` macro allocates a `MaybeUninit<Self>`, calls `sol_get_sysvar` which fills the buffer, then calls `assume_init`. Padding bytes (specified by the `$padding` literal) are zeroed before the syscall to ensure full initialization. On non-SBF targets, `write_bytes(0)` initializes the entire buffer.

Both `Rent` and `Clock` use `impl_sysvar_get!` with padding `0`. The types use alignment-1 Pod fields:

```rust
// core/src/sysvars/rent.rs

pub struct Rent {
    lamports_per_byte: PodU64,    // alignment 1
    exemption_threshold: [u8; 8], // alignment 1
}
const _ASSERT_STRUCT_LEN: () = assert!(size_of::<Rent>() == 16);
const _ASSERT_STRUCT_ALIGN: () = assert!(align_of::<Rent>() == 1);

// core/src/sysvars/clock.rs
#[repr(C)]
pub struct Clock {
    pub slot: PodU64,                  // alignment 1
    pub epoch_start_timestamp: PodI64, // alignment 1
    pub epoch: PodU64,                 // alignment 1
    pub leader_schedule_epoch: PodU64, // alignment 1
    pub unix_timestamp: PodI64,        // alignment 1
}
const _ASSERT_STRUCT_LEN: () = assert!(size_of::<Clock>() == 40);
const _ASSERT_STRUCT_ALIGN: () = assert!(align_of::<Clock>() == 1);
```

Both structs have alignment 1, making `from_bytes_unchecked` (which casts from `&[u8]` to `&Self`) sound on all targets.

### MaybeUninit Patterns

| Location | Code | Purpose |
|----------|------|---------|
| `cpi/mod.rs:126-133` | `MaybeUninit<[RawCpiAccount; ACCTS]>` | CPI accounts array |
| `cpi/system.rs:24-31` | `MaybeUninit<[u8; 52]>` | create_account instruction data |
| `remaining.rs:141` | `MaybeUninit<[AccountView; 64]>` | Iterator duplicate resolution cache |
| `entrypoint.rs:24-30` | `MaybeUninit<[AccountView; COUNT]>` | Declared accounts buffer |
| `sysvars/mod.rs:41` | `MaybeUninit<Self>` | Sysvar syscall buffer |
| `pda.rs:22-31` | `MaybeUninit<Address>` | PDA derivation output |

**Soundness**: Every `MaybeUninit` usage follows the same pattern: allocate, write all bytes (via loop, syscall, or `copy_nonoverlapping`), then `assume_init`. The key invariant is that all bytes are initialized before `assume_init` is called. Each site is individually verified:

- CPI array: `while i < ACCTS` loop writes exactly `ACCTS` elements
- System instruction data: 4 + 8 + 8 + 32 = 52 bytes written, matching buffer size
- Iterator cache: only `assume_init` is NOT called -- elements are read via `ptr::read` at individually validated positions
- Declared accounts: `parse_accounts` writes `COUNT` elements (delegated to derive macro)
- Sysvar: syscall fills the buffer on success; padding bytes zeroed explicitly
- PDA: `assume_init` only called on success (result == 0)

### Tree Borrows Dependency

Quasar explicitly depends on Tree Borrows semantics for the `& -> &mut` cast pattern. This means:

- **Miri must be run with `-Zmiri-tree-borrows`**. Under the default Stacked Borrows model, the `& -> &mut` casts are flagged as UB.
- The Rust compiler does not currently commit to either model. Tree Borrows is the likely successor to Stacked Borrows, but it is not yet stabilized.
- This is an accepted risk, shared with Pinocchio and other Solana frameworks. The entire ecosystem relies on this pattern for zero-copy mutable account access.

A future major version could eliminate this dependency by wrapping `AccountView`'s pointer in `UnsafeCell`, making the pattern sound under all models.

## Attack Surface and Mitigations

### All-Zero Discriminator Attack

**Attack**: An attacker creates an account with all-zero data (the default for new accounts). If the program uses all-zero discriminators, this uninitialized account passes discriminator validation.

**Mitigation**: All-zero discriminators are banned at compile time. The `#[account]` macro generates:

```rust
const _: () = assert!(
    // At least one discriminator byte must be non-zero
    discriminator_byte_0 != 0 || discriminator_byte_1 != 0 || ...
);
```

Since uninitialized account data is all-zero and no valid discriminator is all-zero, uninitialized data can never match a valid account.

### Re-Initialization Attack

**Attack**: An attacker calls an init instruction on an account that already contains valid data, overwriting it.

**Mitigation**: The `init()` method (generated by `#[account]`) checks that the discriminator region is all-zero bytes before writing:

```rust
// Generated init() pseudocode
let disc_region = &data[..DISC_LEN];
if disc_region.iter().any(|b| *b != 0) {
    return Err(QuasarError::AccountAlreadyInitialized);
}
```

Combined with the all-zero discriminator ban, this creates a clean invariant: a valid account always has a non-zero discriminator, and `init()` only succeeds on accounts with zero discriminators (uninitialized).

### Account Revival Attack

**Attack**: After closing an account within a transaction, a subsequent instruction in the same transaction re-opens the account. If the data was not zeroed during close, the old discriminator is still present, and the account passes validation.

**Mitigation**: `Account::close()` (`accounts/account.rs:128`) zeros the discriminator bytes before draining lamports:

```rust
pub fn close(&self, destination: &AccountView) -> Result<(), ProgramError> {
    let view = self.to_account_view();

    // Zero discriminator bytes to prevent revival within the same transaction
    let zero_len = view.data_len().min(8);
    if zero_len > 0 {
        unsafe {
            core::ptr::write_bytes(view.data_ptr(), 0, zero_len);
        }
    }

    destination.set_lamports(destination.lamports() + view.lamports());
    view.set_lamports(0);
    unsafe { view.assign(&SYSTEM_PROGRAM_ID) };
    view.resize(0)?;
    Ok(())
}
```

The discriminator is zeroed first, then lamports are drained, then the owner is reassigned to the System Program. This order ensures that even within the same transaction, the account cannot pass discriminator validation after close.

### Misaligned Access

**Attack**: Account data from the SVM has alignment 1 (it is a byte buffer). Casting to a struct with alignment > 1 produces undefined behavior, even if the data is never read through the misaligned reference.

**Mitigation**: All zero-copy types use Pod fields with alignment 1:

| Rust Type | ZC Type | Alignment |
|-----------|---------|-----------|
| `u64` | `PodU64` (`#[repr(transparent)]` over `[u8; 8]`) | 1 |
| `u128` | `PodU128` (`#[repr(transparent)]` over `[u8; 16]`) | 1 |
| `i64` | `PodI64` | 1 |
| `bool` | `PodBool` (`#[repr(transparent)]` over `u8`) | 1 |
| `Address` | `[u8; 32]` | 1 |

Every ZC companion struct has a compile-time assertion: `assert!(align_of::<ZcStruct>() == 1)`. This applies to:
- Account state ZC structs (generated by `#[account]`)
- Instruction argument ZC structs (generated by `#[instruction]`)
- Sysvar structs (`Rent`, `Clock`)
- `RawCpiAccount` is alignment 8, but it is constructed on the stack, not cast from byte data

### Uninitialized Data Access

**Attack**: Reading account data before validating its length could access memory past the end of the account's data buffer.

**Mitigation**: `AccountCheck::check()` validates `data_len >= required_size` before any pointer cast. This check runs during `from_account_view` / `from_account_view_mut`, which is the only way to construct an `Account<T>`. The subsequent `Deref` impl relies on this upstream check and performs no bounds validation itself.

For remaining accounts, every pointer dereference is preceded by a boundary check (`ptr >= boundary`). The boundary pointer marks the end of the accounts region in the SVM buffer.

### Duplicate Account Chain Overflow

**Attack**: Malformed SVM input data with chained duplicate account references could cause excessive iteration.

**Mitigation**: `resolve_dup_walk` (`remaining.rs:86`) uses iterative resolution with a depth limit of 2:

```rust
fn resolve_dup_walk(...) -> AccountView {
    let mut idx = orig_idx;
    for _ in 0..2 {
        if idx < declared.len() {
            return unsafe { core::ptr::read(declared.as_ptr().add(idx)) };
        }
        // walk to target, check borrow_state
        // if NOT_BORROWED: return AccountView
        // else: idx = borrow as usize; continue (iterative, not recursive)
    }
    unreachable!("duplicate chain exceeded maximum depth")
}
```

The SVM guarantees duplicates resolve in one hop. The depth limit of 2 provides defense-in-depth without risking stack overflow on the 4KB SBF stack.

The iterator's `resolve_dup` method (`remaining.rs:158`) uses a hard bounds check:

```rust
if remaining_idx >= self.index {
    return None;
}
```

This prevents reading uninitialized cache memory for forward-referencing duplicate indices.

## Design Choices

### Explicit Discriminators

Quasar uses developer-specified integer discriminators instead of sha256-derived hashes:

```rust
#[account(discriminator = 1)]
pub struct EscrowAccount { ... }

#[instruction(discriminator = 0)]
pub fn make(...) { ... }
```

Benefits:
- Discriminators are visible in the source code -- no tooling needed to decode them
- Single-byte discriminators save 7 bytes per account and per instruction compared to 8-byte sha256 truncations
- No runtime hashing cost
- Collision detection at compile time (same-kind collisions are compile errors)

Tradeoff: Developers must manually manage discriminator assignments. The compile-time collision check catches mistakes.

### Tree Borrows Dependency

The `& -> &mut` cast pattern is the foundation of zero-copy mutable access in Quasar (and Pinocchio, and Anchor's internals). This pattern is sound under Tree Borrows but UB under Stacked Borrows.

The Rust team has indicated that Tree Borrows (or a model at least as permissive) is the likely future standard. The entire Solana ecosystem depends on this pattern, which provides strong practical assurance that it will not be broken by future compiler versions.

If the Rust aliasing model changes in a way that invalidates this pattern, the fix is to wrap `AccountView`'s inner pointer in `UnsafeCell`, which would make the pattern formally sound under any model at the cost of zero runtime overhead.

### Component Crate Dependencies

Quasar depends on decomposed `solana-*` component crates instead of the monolithic `solana-program`:

- `solana-address` (Address type)
- `solana-account-view` (AccountView, RuntimeAccount)
- `solana-instruction-view` (InstructionAccount, CPI types)
- `solana-program-error` (ProgramError)
- `solana-define-syscall` (syscall definitions)

This reduces compile times and dependency surface. The tradeoff is tighter coupling to the SVM's internal serialization layout (e.g., `RuntimeAccount` field order, buffer structure), which is guaranteed stable by the Solana runtime but not by a formal specification.

### Zero Heap Allocation

The `no_alloc!()` macro installs a global allocator that panics on any heap allocation:

```rust
// core/src/entrypoint.rs:46-66
pub struct NoAlloc;
unsafe impl GlobalAlloc for NoAlloc {
    unsafe fn alloc(&self, _: Layout) -> *mut u8 {
        panic!("");
    }
    unsafe fn dealloc(&self, _: *mut u8, _: Layout) {}
}
```

This is a compile-time and runtime guarantee that the entire dispatch-parse-CPI path is zero-allocation. Any accidental `Vec`, `String`, or `Box` usage triggers a panic during testing. The `heap_alloc!()` macro provides the standard Solana bump allocator for programs that need heap access (e.g., for logging or complex client-side operations).
